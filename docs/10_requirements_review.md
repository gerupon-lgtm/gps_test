## 1. 現状整理

現在の検証版では、位置情報を利用したスポット判定、戦闘、報酬、ユーザー状態管理、宿屋、道具屋、マーケット、レベル、毒、散策取得など、ゲーム基盤となる多くの要素が実装・検証されている。戦闘では攻撃・アイテム使用に加えて「にげる」コマンドが追加され、逃走時は報酬・撃破履歴・戦闘不能ペナルティなしで戦闘を終了する。マーケットでは、HUDからの出品・購入・取消、売買確認、固定5G+割合0%を初期値とする手数料、手数料台帳、他プレイヤー出品時の低優先トースト通知が追加されている。スポット一覧では、ゲーム内表現に合わせて「撃破済み」バッジを表示し、クールダウン中は残り分数も併記する。

加えて、DBを正本とする管理運用の検証が進み、`game-admin/` にDB管理Webアプリが追加された。管理アプリでは、管理者ログイン、無操作タイムアウト、バージョン表示、再ログイン時の画面初期化、プレイヤー確認、ログイン停止、セッション削除、撃破済みスポット編集、マスタ一覧/詳細編集、マスタのCSV出力・CSV取り込み、取り込みプレビュー、近接チェック、1件ずつの新規登録、操作ログ記録が実装されている。

また、`入力データ作成/` に施設名リストからGoogle Maps Geocoding APIで座標を取得し、国土地理院の逆ジオコーディングAPIで市区町村コード等を補助取得するローカルGUIツールが追加された。スポット・宿屋・道具屋候補の初期データ作成を補助する位置づけ。

一方で、ゲームとしての最終目的、スポット登録ルールの最終判断、パラメータ上限、想定ユーザー数、想定スポット数、実コスト、サーバー移行後の運用体制などは、今後の判断事項として残っている。後続候補として、ランダム出現NPC、NPCからのクエストヒント、行商人、少量回復、他プレイヤーとの遭遇通知も検討対象に加える。

本資料では、検討事項を以下に分けて整理する。

- 機能要件: ユーザーや管理者が利用する機能・ゲーム仕様に関する要件
- 非機能要件: 性能、セキュリティ、運用、コスト、保守性、安全性などに関する要件
- 未決事項: 方針決定が必要な論点
- 優先度: 本番化前に必須か、後回しにできるか

### 1.1 現時点で実装・検証が進んだもの

| 分類 | 現状 |
|---|---|
| ゲーム本体 | GPS、地図、認証、サーバー権威のターン制戦闘、報酬、レベル、毒、戦闘不能、散策取得、宿屋、道具屋、マーケット |
| DB管理 | Prisma/PostgreSQLを正本とし、Prisma Studioによる一時メンテナンス手順を整備 |
| 管理Webアプリ | `game-admin/` として別アプリ化。想定URLは `https://admin.gps.gerupon.uk/` |
| プレイヤー管理 | プレイヤー一覧/詳細、ログイン停止/解除、セッション削除、撃破済みスポット編集 |
| マスタ管理 | スポット、敵、アイテム、宿屋、道具屋、郵便番号/地域の一覧/詳細編集、active切替、既存レコードからのコピー登録 |
| CSV入出力 | 管理アプリからCSV出力/取り込み、SJIS/UTF-8対応、行ごとの取込可否、画面上編集、プレビュー、upsert反映 |
| 近接チェック | スポット/宿屋/道具屋の取り込み時・新規登録時・既存データに対して近接警告を表示 |
| 操作監査 | `AdminAuditLog` に管理操作を記録 |
| 入力補助 | 施設名リストから座標・地域情報を取得するローカルGUIツールを追加 |
| マーケット | HUDメニューから出品/購入/取消。道具屋買取価格を初期値にし、手数料負担を売り手/買い手から選択 |
| スポット一覧 | 近い順の一覧に「撃破済み」バッジを表示。クールダウン中は残り分数も表示 |
| 戦闘逃走 | 戦闘中に「にげる」を選択し、報酬・ペナルティなしで戦闘終了 |

## 2. 機能要件

### 2.1 ゲーム目的・クリア要素

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| クリア条件 | プレイヤーが何を達成したら一区切りとするかを定義する | 高 | 地域制覇、重要スポット制覇、特定クエスト達成、ラスボス撃破、図鑑完成など |
| 進行状況表示 | クリアに向けた達成率を表示する | 中 | 地域別達成率、重要スポット数、クエスト達成数 |
| エンディング・達成演出 | クリア時の表示や報酬を用意する | 低 | 検証段階では後回し可 |
| 継続プレイ要素 | クリア後も遊べる要素を定義する | 低 | 周回、追加地域、ランキング、収集要素 |

現時点ではゲーム目的が未定のため、まずは「地域・スポットを巡る理由」を定める必要がある。技術検証の次段階では、最小構成として「重要スポットを一定数攻略するとクリア」とする案が扱いやすい。

### 2.2 スポット種別

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| 通常スポット | 通常の敵出現・報酬取得の対象地点 | 高 | 現行の `SpotMaster` を基本に継続 |
| 重要スポット | クリア条件やイベントに関係する特別な地点 | 高 | 通常スポットと同じDBで type を持たせるか、別マスタにするか |
| イベントスポット | 期間限定・条件付きで出現する地点 | 中 | 有効期間、公開範囲、イベント報酬 |
| 宿屋 | HP回復、毒解除、戦闘不能解除を行う地点 | 実装済 | 実施設座標への差し替えが必要 |
| 道具屋 | アイテム売買を行う地点 | 実装済 | 実施設座標への差し替え、数量選択。販売可否は `shopBuyable`、売却可否は `sellable` で制御 |
| 所持金表示 | 宿屋、道具屋、マーケットで所持金を表示する | 実装済 | 所持金不足の商品・出品は非活性にする |
| NPC | マップ上に出現し、会話・ヒント・回復・販売などを行うキャラクター | 中 | 施設とは別マスタ推奨。ランダム出現、表示期間、再出現、効果、会話内容を管理 |
| ランダム出現NPC | 条件に応じて一時的に出現するNPC | 中 | 出現確率、出現範囲、出現時間、ユーザー共通/個別、再抽選タイミング |
| 行商人NPC | 固定道具屋とは別に、限定商品を販売するNPC | 中 | 通常道具屋と価格/在庫/品揃えを分ける。購入処理はサーバー権威 |
| 回復NPC | 少量HP回復などを行うNPC | 中 | 宿屋との差別化。毒解除・戦闘不能解除を許可するか要判断 |
| 非表示スポット | 地図上には見せず、近づいたときだけ反応する地点 | 中 | 現行の敵スポット方針に近い |

通常スポットと重要スポットは、ゲーム目的が決まるまで分け方を確定しにくい。暫定的には `spot_type` のような区分をマスタに追加できる設計にしておくのがよい。

### 2.3 スポット登録ルール

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| 登録対象の基準 | 登録してよい場所、避ける場所を定義する | 高 | 公共性、安全性、立ち入り可否、私有地除外 |
| 座標精度の基準 | 登録座標の精度・確認方法を決める | 高 | Google Maps等で確認、現地確認、誤差許容 |
| スポット間距離 | 近すぎるスポットの登録可否を決める | 中 | 例: 半径50m以内の重複を避ける |
| 出現半径 | スポットごとの判定半径の標準値を決める | 高 | 30m/50m/100m。GPS精度との兼ね合い |
| 危険地点除外 | 交通量が多い場所、線路、河川、私有地等を除外する | 高 | 安全性・責任範囲に直結 |
| 公開停止 | 問題があるスポットを無効化できる | 高 | `active=false` の運用、即時反映方法 |
| 実地点確認 | 現在の概算宿屋・道具屋座標を実施設座標へ置換する | 高 | 入力データ作成ツールと管理アプリの近接チェックを活用 |
| 近接警告 | 登録・取り込み時に既存施設との距離を確認する | 実装済 | 既定10m。警告のみで反映は止めない |
| 初期データ作成 | 施設名リストから座標・地域情報をCSV化する | 実装済 | `入力データ作成/app.py`。Google Maps APIキーが必要 |

スポット登録はゲーム性だけでなく安全性・運用責任に直結するため、本番前の重要検討事項とする。ツール面では登録・取り込み・近接チェックの土台ができたため、次は「登録してよい場所/避ける場所」の判断基準を文書化する段階。

### 2.4 クエスト・出題要素

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| クエスト | 特定スポット訪問、敵撃破、アイテム収集などの課題を提供する | 中 | クリア要素と連動させるか |
| NPCヒント | NPCが後続クエストのヒントや誘導文を表示する | 中 | クエスト本体より先に、会話だけの軽量実装も可能 |
| スポット連動出題 | 勝利時や重要スポット到達時に、スポットに関連した「はしけん」等の問題を出す | 中 | 正解報酬、不正解時の扱い、再挑戦 |
| クエスト進行管理 | ユーザーごとの進行状態を保存する | 中 | DBモデル追加が必要 |
| 報酬設計 | クエスト達成時の経験値・ゴールド・アイテムを定義する | 中 | 既存報酬とのバランス |

クエストはゲーム目的が決まった後に設計した方がよい。先に実装すると、目的未定のまま報酬や進行管理だけが増える可能性がある。

### 2.5 パラメータ・上限値

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| レベル上限 | プレイヤーレベルの上限を定義する | 高 | 無制限だと数値肥大・バランス崩壊の恐れ |
| HP/攻撃/防御上限 | ステータス最大値を定義する | 高 | DB上限、UI表示幅、戦闘計算 |
| 経験値上限 | 累積EXPの上限や必要EXP曲線を定義する | 中 | 現在は `level × LEVEL_EXP_FACTOR` |
| ゴールド上限 | 所持金の最大値を決める | 中 | マーケットや売買の悪用防止 |
| アイテム所持上限 | アイテム種類・数量の上限を決める | 中 | DB肥大化、UI表示、取引 |
| 出品上限 | マーケットの同時出品数・価格上限を決める | 中 | 荒らし・DB肥大化防止 |
| マーケット手数料 | 出品購入時・取消時の手数料を設定できる | 実装済 | 初期値は固定5G+割合0%。売買時は売り手/買い手負担を選択、取消時は売り手負担 |
| 戦闘セッション制限 | 同時戦闘数は1のままにする | 実装済 | 現行 `BattleSession.playerId @unique` |

現段階では検証値で動いているため、本番化前に「上限値一覧」と「設定変更方法」を資料化する必要がある。

### 2.6 マスタ管理

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| マスタ編集 | スポット、敵、アイテム、宿屋、道具屋、郵便番号/地域を編集できる | 実装済 | `game-admin/` のマスタ詳細編集。コピーして新規登録も可能 |
| マスタ反映 | DBを正本として、管理アプリから追加/更新する | 実装済 | CSV未掲載レコードは削除しないupsert方式 |
| CSV出力 | 現在のDB内容をCSVとして取得する | 実装済 | SJIS/UTF-8対応 |
| CSV取り込み | CSVをプレビューして追加/更新を反映する | 実装済 | 行ごとに取込可否を切替可能。エラーがある場合は反映不可 |
| 1件登録 | 画面から1件ずつマスタ追加する | 実装済 | ID空欄時は自動採番。参照IDや分類項目は候補選択に対応 |
| バリデーション | 型、ID重複、必須項目、数値項目を検証する | 実装済 | 参照整合性の強化は今後検討 |
| 近接チェック | スポット/宿屋/道具屋の近接警告を出す | 実装済 | 既存データ同士の確認も可能。近接行は既定で取込OFF |
| 管理権限 | 管理者ログインを必須にする | 実装済 | `ADMIN_USER` / `ADMIN_PASSWORD`。無操作タイムアウトは既定600秒 |
| 変更履歴 | 管理操作ログを保存する | 実装済 | `AdminAuditLog` |
| 差分プレビュー | CSV取り込み前に追加/更新/変更なし/未掲載/エラーを確認する | 実装済 | 重複/近接警告表示、booleanチェックボックス、画面上編集に対応 |

本番前の最小構成として、DBを正本にした管理Webアプリが使える状態になった。今後は、参照整合性チェック、運用権限の分離、誤操作時のロールバック、管理アプリ自体のアクセス制限を詰める。

### 2.7 ユーザー・セーブ・アカウント

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| ログイン | ユーザーが自分のデータを継続利用できる | 実装済 | Cookieセッション |
| 招待制登録 | 想定外ユーザーの流入を制限する | 高 | 検証中は特に有効 |
| セーブデータ | プレイヤー状態、所持品、進行状況をDBに保存する | 実装済 | サーバー権威 |
| ログイン停止 | 問題があるユーザーを停止できる | 実装済 | `User.disabled`。停止時はセッション削除 |
| セッション削除 | ユーザーのログイン状態を管理側から破棄できる | 実装済 | 管理アプリから実行 |
| 撃破済み編集 | 個別プレイヤーの撃破済みスポットを修正できる | 実装済 | `defeatedSpots` と `PlayerSpotState.victoryUntil` を同期 |
| アカウント復旧 | パスワード忘れ等への対応 | 低 | 小規模検証では手動対応でも可 |
| 退会・削除 | ユーザー削除の方針を決める | 中 | 個人情報・データ保持方針 |

### 2.8 位置・ソーシャル要素

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| 現在地判定 | GPS精度・距離でスポット近接判定する | 実装済 | サーバー側の厳密検証は現状一部方針緩め |
| 近くのプレイヤー | 位置共有ONのユーザーを近距離表示する | 低 | `shareLocation` / `lastSeenAt` はスキーマあり |
| プレイヤー遭遇通知 | 他プレイヤーと一定距離以内に近づいた時に「○○にであった」をトースト表示する | 中 | 通知しきい値、再通知クールダウン、位置共有ONのみ対象、古い位置情報除外 |
| 位置保存方針 | 位置情報を永続保存するかを決める | 高 | プライバシーに直結 |
| 位置偽装対策 | どこまで対策するかを決める | 中 | 小規模なら深追いしない方針も可 |

現方針では、位置偽装対策は深追いしない。公開範囲が広がる場合は、異常移動速度検知やレート制限程度は検討する。

## 3. 非機能要件

### 3.1 インフラ・サーバー移行

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| 本番サーバー | 検証サーバーとは別インスタンスで運用する | 高 | OCI/VPS等。移行手順が必要 |
| 環境分離 | 検証環境と本番環境を分ける | 高 | DB、ドメイン、環境変数 |
| HTTPS | 常時HTTPSで提供する | 高 | Caddy / Let's Encrypt |
| DB | PostgreSQLを継続利用する | 高 | バックアップ・接続数制限 |
| 管理アプリ | ゲーム本体とは別サブドメインで管理Webアプリを運用する | 実装中 | 想定 `admin.gps.gerupon.uk`、127.0.0.1:3010へリバースプロキシ |
| Prisma Studio | 一時メンテナンス用にSSHトンネル経由で利用する | 手順化済 | 恒久運用は管理Webアプリを正とする |
| デプロイ手順 | API、フロント、データ更新の手順を明文化する | 高 | 現行手順の標準化 |
| ロールバック | 問題発生時に前バージョンへ戻す方法を用意する | 中 | git tag、DBバックアップ |

サーバー移行時は、データ移行、環境変数、ドメイン、HTTPS、systemd、DBバックアップをまとめた移行チェックリストが必要。

### 3.2 コスト

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| VPS/クラウド費用 | API/DBを動かすサーバー費用 | 高 | OCI Always Free継続か、有償VPSか |
| 地図タイル費用 | MapTiler等の利用料 | 高 | 無料枠、APIキー制限、超過時料金 |
| 独自ドメイン | ドメイン取得・更新費用 | 中 | 年額費用 |
| バックアップ保管 | DBバックアップ保存先の費用 | 中 | オブジェクトストレージ等 |
| 監視 | 死活監視・ログ監視ツール費用 | 低 | 小規模なら無料枠でも可 |

想定ユーザー数と地図表示回数によって MapTiler 費用が変わるため、本番前に「月間アクセス数」「1ユーザーあたり地図表示回数」を仮置きする。

### 3.3 想定規模・性能

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| 想定ユーザー数 | 初期・最大の利用者数を定義する | 高 | 例: 初期10人、拡大時50人など |
| 同時接続数 | 同時に操作するユーザー数を定義する | 高 | DB接続数、API負荷 |
| 想定スポット数 | 初期・最大スポット数を定義する | 高 | 現状は小規模。数百以上で検索方式検討 |
| 想定NPC数 | 固定NPC・ランダム出現NPCの最大数を定義する | 中 | 地図マーカー数、出現判定API、管理画面表示負荷 |
| レスポンス時間 | 主要APIの体感速度目標を決める | 中 | 例: 通常操作1秒以内 |
| 地図表示性能 | マーカー数増加時の表示負荷を考慮する | 中 | 敵スポット非表示方針は有効 |
| 出現判定負荷 | ランダム出現NPCや遭遇通知の判定頻度を制御する | 中 | 位置更新ごとの全件判定は避ける。近傍検索、クールダウン、ポーリング間隔を設定 |
| DB接続数 | Prisma接続プールを小さく保つ | 高 | 小規模VMでは重要 |

スポット数が数十〜数百なら現方式でも扱いやすい。数千規模を想定するなら、スポット検索・地理インデックス・マスタ配信方式の見直しが必要。

### 3.4 セキュリティ

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| サーバー権威 | 戦闘、報酬、通貨、取引をサーバーで確定する | 高 | 現行方針を維持 |
| 認証 | Cookieセッションを安全に扱う | 高 | HttpOnly, Secure, SameSite |
| パスワード保存 | ハッシュ化して保存する | 高 | 現行は scrypt 系 |
| 入力検証 | API入力を検証する | 高 | zod等。現行実装確認が必要 |
| レート制限 | ログイン・API連打を制限する | 中 | 小規模でも最低限あると安心 |
| NPC効果のサーバー権威 | NPC会話以外の回復・購入・報酬はサーバーで確定する | 高 | クライアント距離判定だけで回復/購入を許可しない |
| ランダム出現の不正対策 | NPC出現抽選や遭遇判定を改ざんされにくくする | 中 | 出現結果をサーバー管理にする。クライアント側だけの乱数にしない |
| 秘密情報管理 | DB URL、SESSION_SECRET等をコミットしない | 高 | `.env` / systemd |
| 管理機能保護 | 管理Webアプリを管理者ログインで保護する | 実装済 | 追加でCloudflare等のアクセス制限を検討 |
| 管理監査 | 管理操作をログに残す | 実装済 | `AdminAuditLog` |

### 3.5 プライバシー・法務・規約

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| 位置情報方針 | 位置情報の利用目的・保存有無を明示する | 高 | 本番公開時は必須 |
| 位置情報保存 | 直近位置を保存するか、判定のみで破棄するか | 高 | 近くのプレイヤー機能と競合 |
| 位置共有同意 | 他プレイヤー遭遇や近くのプレイヤー表示は明示的にONのユーザーだけ対象にする | 高 | `shareLocation=true` のみ対象。既定OFFを推奨 |
| 位置情報の鮮度 | 古い位置情報を遭遇判定に使わない | 高 | `lastSeenAt` が一定時間以内のデータだけ使用。無操作時は対象外 |
| 利用規約 | 禁止事項、免責、位置情報利用を記載する | 中 | 共有範囲が広がるなら必要 |
| プライバシーポリシー | 取得情報、保存期間、削除方法を記載する | 中 | 位置情報を扱うため優先度高め |
| 地図利用規約 | MapTiler等の規約に沿う | 高 | APIキー制限、著作権表記 |

身内検証では簡易でもよいが、他者へ共有する段階では「位置情報を扱うサービス」としての説明文を用意する。

### 3.6 安全性

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| 歩きスマホ抑止 | 移動中操作を促さないUIにする | 高 | 注意表示、戦闘操作タイミング |
| 危険地点除外 | 危険・立入不可・交通量が多い地点を登録しない | 高 | スポット登録ルールに含める |
| 夜間利用配慮 | 夜間に危険な場所へ誘導しない | 中 | イベント時間制限など |
| NPC出現位置の安全性 | ランダムNPCが危険地点や立入不可地点に出ないようにする | 高 | 完全ランダム座標ではなく、安全確認済みスポット/エリア内から出現させる |
| 子ども利用想定 | 年齢層を想定する | 低 | 身内検証なら後回し |
| 緊急停止 | 問題スポット・イベントを即時停止できる | 高 | active=false、管理者操作 |

ゲーム性より先に、安全上避けるべき地点のルールを決める必要がある。

### 3.7 運用・保守

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| バックアップ | DBを定期バックアップする | 高 | `pg_dump`、保存先、復元手順 |
| 監視 | API/DBの死活を監視する | 中 | `/api/health` 活用 |
| ログ | エラー・重要操作を確認できる | 中 | systemd journal、アプリログ |
| 問い合わせ対応 | ユーザーからの不具合報告を受ける導線 | 低 | 小規模なら直接連絡でも可 |
| データ修正 | 誤付与・不具合時にユーザーデータを修正できる | 実装済 | 管理アプリで一部対応。Prisma Studioは一時対応 |
| バージョン管理 | デプロイしたバージョンを追跡する | 実装中 | 管理アプリはヘッダーにバージョン表示。全体リリースはgit tag、リリースメモも併用 |

### 3.8 保守性・開発体制

| 要件 | 内容 | 優先度 | 検討ポイント |
|---|---|---:|---|
| 設定値管理 | 戦闘・報酬・距離などを環境変数またはマスタで管理する | 高 | 現行 `.env` を整理 |
| NPC出現設定管理 | NPC出現率、表示時間、再出現クールダウン、効果量を設定化する | 中 | マスタ化推奨。運用中に調整できるようにする |
| テスト | 取引、戦闘、報酬、認証の主要処理をテストする | 中 | 既存 trade-concurrency を拡張 |
| ドキュメント更新 | 実装と仕様のズレを減らす | 高 | as-built を正にする運用 |
| CSV/API二重管理解消 | マスタ配信を一本化する | 進行中 | 管理アプリはDB正本。ゲーム側の参照経路整理が残課題 |
| 実装範囲管理 | ゲーム目的決定前に機能を増やしすぎない | 高 | 検証と本番要件を分ける |

## 4. 未決事項一覧

| 分類 | 未決事項 | 判断が必要な内容 | 推奨初期方針 |
|---|---|---|---|
| ゲーム目的 | クリア要素 | 何を達成したらクリアか | 重要スポット制覇を仮置き |
| ゲーム目的 | 重要スポット | 通常スポットとの差分 | `spot_type` 追加を検討 |
| クエスト | はしけん出題 | どのタイミングで出題するか | 重要スポット勝利後に限定 |
| NPC | ランダム出現 | 固定位置か、一時出現か、ユーザー共通か個別か | 安全確認済み地点/エリアから抽選し、出現結果はサーバー管理 |
| NPC | NPC機能 | ヒント、行商人、少量回復、クエスト導線のどこまで初期実装するか | まずはヒント+少量回復、次に行商人 |
| ソーシャル | プレイヤー遭遇通知 | 何m以内で通知するか、再通知間隔、対象ユーザー | 位置共有ONかつ直近位置が新しいユーザーのみ対象 |
| スポット | 登録ルール | 安全性・公共性・重複基準 | 先に文書化する |
| スポット | 想定スポット数 | 初期・最大件数 | 初期100件以内を仮置き |
| ユーザー | 想定ユーザー数 | 初期・最大人数 | 初期10〜30人程度を仮置き |
| パラメータ | 上限値 | Lv/HP/攻撃/防御/所持数 | 本番前に上限一覧を作る |
| 運用 | マスタ管理 | 管理アプリを本番運用の正とするか | DB正本 + 管理アプリを基本方針にする |
| 運用 | 管理アプリ公開 | adminサブドメインをどう保護するか | 強パスワード + 可能ならCloudflare等で制限 |
| 運用 | CSV運用 | DB正本後のCSVの位置づけ | 入出力・初期投入・一括更新用に限定 |
| インフラ | サーバー移行 | どのインスタンスへ移すか | 検証/本番を分離 |
| コスト | 実コスト | MapTiler/VPS/ドメイン等 | 月額見積もり表を作る |
| セキュリティ | 位置偽装 | どこまで対策するか | 小規模では深追いしない |
| プライバシー | 位置保存 | 位置を保存するか | 原則保存しない。共有機能だけ任意 |
| プライバシー | 遭遇通知 | 他プレイヤー名を通知してよい条件 | 明示的に共有ONのユーザーだけ通知対象 |

## 6. コスト検討メモ

| 項目 | 検討内容 | 備考 |
|---|---|---|
| VPS/クラウド | API/DB常時稼働の費用 | OCI Always Free継続か、有償VPSかを比較 |
| DB | PostgreSQLの保存容量・バックアップ | 同一VM運用なら障害時の復元が課題 |
| MapTiler | 地図タイル表示回数・無料枠・超過料金 | ユーザー数と地図利用頻度で変動 |
| 独自ドメイン | 年額費用 | DNS管理も含める |
| SSL | Let's Encryptなら無料 | 自動更新の監視が必要 |
| バックアップ保存 | 外部保存先の費用 | 無料枠または手動保存でも可 |
| 監視 | 死活監視・通知 | 小規模なら無料サービスで開始可能 |

月額見積もりを作る場合は、以下を仮置きする。

- 月間アクティブユーザー数
- 1人あたりの平均プレイ日数
- 1日あたりの地図表示回数
- 1日あたりのAPI操作回数
- スポット数
- DBバックアップ保持期間
