# リズム戦闘PoC 時計・描画診断モード設計

作成日: 2026-06-20

## 目的

同一Android端末の通常版Chromeで、端末音量0のときだけ発生する「描画が徐々に遅れ、最後はタップを受け付けなくなる」症状を切り分ける。通常プレイの動作は変更せず、原因確認に必要な値だけを画面へ表示する。

## 有効化

- URLへ`?debug=1`を付けた場合だけ診断モードを有効にする。
- 通常URLでは診断DOMを非表示にし、フレーム計測とDOM更新を行わない。
- 配布版`dist/rhythm-battle-poc/index.html?debug=1`でも同じように利用できる。

## 表示項目

- `navigator.userAgent`の短縮表示
- `AudioContext.state`
- 音声時計による曲内経過秒
- `performance.now()`による曲内経過秒
- 終了時および最大の実時間と音声時計の差`driftMs`
- 最大フレーム間隔
- 50ms以上空いたフレーム数
- `baseLatency`と、利用可能な場合は`outputLatency`
- 終了理由、`running`、最後のタップ判定
- 戦闘中に発生したAudioContext状態変化

## 計測方法

- 戦闘開始時に、`state.startTime`へ対応する`performance.now()`上の曲開始時刻を保存する。
- 描画フレームごとに音声曲内時刻と実時間曲内時刻を算出し、数値だけをメモリ上へ集計する。
- `driftMs = (wallSongTime - audioSongTime) * 1000`とする。正の値が増え続ける場合、音声時計が実時間より遅れている。
- フレーム間隔は`requestAnimationFrame`内の`performance.now()`差分から求める。
- プレイ中は診断パネルを非表示とし、DOM更新を行わない。計測自身による再レイアウト・再描画を避ける。
- `AudioContext`の`statechange`を監視し、`suspended`、`interrupted`、`closed`への変化を表示する。
- 撃破または時間切れで再生を停止した後に一度だけ最終集計を表示する。

## 表示位置と操作性

- 終了後のパネルはレーン内左上へ半透明・等幅フォントで表示する。
- `pointer-events: none`とし、レーンおよび「こうげき」の入力を妨げない。
- 診断中は表示情報を優先するため、ノーツと一部重なっても許容する。
- 戦闘終了後に最終値を表示して残し、落ち着いてスクリーンショットできるようにする。

## 判定基準

- 最大`driftMs`と終了時`driftMs`が大きい: Android Chromeの音声時計と実時間の乖離が主因。
- `driftMs`は小さいが最大フレーム間隔と50ms超過数が増加: メインスレッド描画負荷が主因。
- `AudioContext.state`が`running`以外へ変化: Chromeの音声コンテキスト状態変化が主因。
- `running=false`になった時点で音声曲内時刻が曲長未満: 壁時計の曲終了タイマーとの不一致がタップ停止の直接原因。

## 対象外

- 時計の切り替え、曲終了条件、音源生成の最適化
- 判定幅、譜面、スコア、敵HPの変更
- 診断結果を外部へ送信または永続保存する処理

修正は診断結果を確認した後、原因に対応する1点だけを別作業として実施する。

## テスト

- 時計差計算の純粋関数をNode.jsテストで確認する。
- `?debug=1`判定、診断パネル、`pointer-events: none`、プレイ中にDOM更新しないことを静的テストで確認する。
- 撃破・時間切れの両方で終了後サマリーを表示することを確認する。
- 通常URLでは診断更新処理が早期終了することを確認する。
- 既存45テストとJavaScript構文確認を維持する。
