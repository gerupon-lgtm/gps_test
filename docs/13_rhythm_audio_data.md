# リズム戦闘PoC 音声・譜面・視覚プレイ仕様

最終更新: 2026-06-20

本書をリズム戦闘PoCの正本とする。GPSゲーム本体へリズム戦闘を導入できるか検証するため、Web Audio APIだけで動作する独立HTML版PoCを対象に、音楽と判定の同期、スマートフォン操作、端末遅延補正、譜面別バランス、無音環境を含む視覚プレイアビリティをまとめる。

PoCはゲーム本体へ未統合であり、バックエンド、DB、ログイン、位置情報、外部ライブラリ、外部APIに依存しない。

## 1. 推奨方式

撃破済みスポット数に応じてBPMを変える場合は、完成済み音源だけを再生する方式より、Web Audio APIで音符・ドラムなどのイベントを予約する方式が適している。

- イベント方式: BPM変更に追従しやすく、判定ノートと同じ時計を使える。
- MP3/WAV方式: 制作しやすいが、BPM変更には再生速度変更が必要で、音程や音質が変化する。
- 実用上は、固定BPMの音源を複数用意するか、イベント方式と短い効果音素材を組み合わせる方法が扱いやすい。

## 2. 基本データ

曲データは最低限、次の値を持つ。

```json
{
  "id": "battle_001",
  "baseBpm": 126,
  "timeSignature": { "beatsPerBar": 4, "beatUnit": 4 },
  "bars": 8,
  "countInBeats": 4,
  "loop": false,
  "events": [
    { "beat": 0, "type": "kick", "velocity": 1.0 },
    { "beat": 1, "type": "snare", "velocity": 0.8 }
  ],
  "attackNotes": [
    { "beat": 0, "lane": 0, "accent": true }
  ]
}
```

現在のPoCは4/4拍子、8小節、初期値126 BPMである。BPMは80から180まで戦闘開始前に変更でき、戦闘中は固定する。126 BPMでは譜面部分の長さが次の式で約15.24秒、4拍カウントを含めると約17.14秒となる。

PoCでは次の5曲と5譜面を独立して選択できる。

- ストレート・バトル: 均等な8分音符の4/4戦闘曲。
- シンコペーション・ドライブ: 8分裏のベースとコードを強調する曲。
- シャッフル・クエスト: 8分裏を2/3拍位置へずらした跳ねる曲。
- 余白のステップ: 長い音価と少ない打楽器で休符を楽しむ曲。
- ミッドナイト・ブレイク（JAZZ）: 8小節連続の四分音符ランニングベースと、小節頭の全音符ピアノで構成する曲。
- 基本譜面: 四分音符中心。
- 裏拍譜面: 各曲のグルーヴに応じた8分裏中心。
- 技巧譜面: 表、裏、休符、短い連続入力を組み合わせる。
- 余白譜面: 1小節2タップを基本とする低密度譜面。
- ブレイク技巧譜面: 前半を密集させ、5～6小節を休み、7小節目後半から復帰する展開型譜面。

譜面表示は1ボタン操作のまま、表拍を左レーンの金色ひし形、均等8分の裏拍を右レーンの水色丸、シャッフル裏を右レーンの紫色丸で表示する。拍種別による重要度の誤解と密集を避けるため、すべて22px四方へ統一する。アクセントはサイズを変えず、輪郭と発光で区別する。

ミッドナイト・ブレイク（JAZZ）は `Dm7 → G7 → Cmaj7 → A7(b9)` のii–V–I–VIを2回繰り返す。全32拍に四分音符ベースを1音ずつ置き、各小節の4音目から次コードのルートへ半音で接続する。後半4小節は同じ進行でも上行・下行を変える。

ピアノは各小節頭に7thコードを1回だけ発音し、約3.8拍かけて自然減衰させる。後半は転回形を変えて声部を滑らかにつなぐ。ジャズ曲ではピアノ単音フレーズ、途中のコード打鍵、キック、ブラシ、ライド、ハイハット、小節アクセントを使用しない。

ジャズ専用ベースはピーク0.20で、サイン波の基音へ第2倍音36%、第3倍音12%を加え、700Hzのローパスへ通す。ピアノはピーク0.075で、各コード音をサイン波の基音・2倍音・3倍音から合成し、倍音ほど速く減衰させる。

PoCの曲は8小節・32拍で終了し、戦闘が未完了でもループしない。曲終了後は攻撃を無効化し、再度「戦闘開始」を押すまで停止する。戦闘開始時は予約済みのOscillatorNodeとAudioBufferSourceNode、scheduler、各timer、画面animationを停止し、敵HP・譜面・スコアを含めて曲頭からやり直す。敵撃破時も予約済みの残り音を即時停止する。

```text
曲秒数 = 小節数 × 1小節の拍数 × 60 ÷ BPM
8 × 4 × 60 ÷ 126 = 約15.24秒
```

## 3. AudioContextの時刻

`AudioContext.currentTime` はAudioContext生成後から進む読み取り専用の時計であり、外部データから値を代入することはできない。外部曲データは曲内の相対位置を持たせ、実行時にAudioContextの絶対時刻へ変換する。

```js
const songStartTime = audioContext.currentTime + 0.2 + countInDuration;
const eventTime = songStartTime + eventSeconds;
source.start(eventTime);

const currentSongTime = audioContext.currentTime - songStartTime;
```

外部から指定できるのは `songStartTime` の基準、曲内オフセット、BPM、イベント位置である。判定、画面表示、音声予約のすべてが同じ `songStartTime` を参照する。

MP3/WAVを使う場合は `decodeAudioData()` で `AudioBuffer` にし、`AudioBufferSourceNode.start(when, offset)` で開始時刻と曲内位置を指定する。`currentTime` 自体は変更しない。

## 4. BPMと拍子

固定BPMなら、曲内拍位置から秒への変換は次の式でよい。

```text
eventSeconds = beat × 60 ÷ BPM
```

拍子変更だけであれば、イベントを通算拍位置で保持することで対応できる。テンポ変更を行う場合は、次のようなテンポマップを追加し、区間ごとに経過秒数を累積する。

```json
"tempoMap": [
  { "beat": 0, "bpm": 126 },
  { "beat": 32, "bpm": 140 }
]
```

複雑な変拍子・テンポ変更まで扱う場合は、MIDIと同様にPPQ（例: 480 ticks/quarter note）を採用する。通常の4/4、3/4、6/8や数回の拍子変更であれば、通算拍とテンポマップで対応可能である。

## 5. 撃破数によるBPM変更

本体実装時は、戦闘開始前に一度だけBPMを算出して固定する。

```text
battleBpm = min(最大BPM, 基準BPM + floor(撃破済みスポット数 ÷ 5) × 2)
```

戦闘途中でBPMを変更すると、予約済み音声、ノート位置、判定時刻の再計算が必要になるため行わない。曲の基準BPMと適用可能な最小・最大BPMも曲データに持たせる。

## 6. 曲データ受け渡し時の確認項目

- 基準BPMと許容BPM範囲
- 拍子と途中の拍子変更位置
- 小節数、ループ開始・終了位置
- カウント数と曲頭の無音時間
- 音楽イベントと攻撃判定ノートの拍位置
- MP3/WAV素材を使う場合の基準BPM、サンプルレート、ループ点
- 曲終了時にループ、勝敗確定、別フレーズへ移行のどれを行うか

## 7. PoCのスマートフォン表示

- 基準表示は幅390px、高さ844pxとし、幅375px、高さ667pxまでを最低確認対象とする。
- スマートフォンでは曲、譜面、BPM、ヒント、戦闘開始を横一列にまとめ、説明文を省略する。
- 譜面レーンは`100dvh`を基準に高さを計算し、端末の表示領域が広いほどプレイ部分を拡張する。
- BPM、判定、コンボ、スコアは横4列で表示し、戦闘ログは最初の画面より下に配置する。
- 高さ600px以下ではタイトルと敵画像も省略し、譜面と操作領域を優先する。
- ノートの判定位置は固定ピクセル値ではなく、画面上の判定線中央から取得する。レーン高が変わっても音声時刻と表示位置を一致させる。
- 判定窓外のタップはMISS表示のみとし、次のノートを消費しない。入力ミスによる譜面の位相ずれを防ぐ。
- 曲は8小節で終了し、未撃破の場合は停止したままとする。「戦闘開始」で戦闘状態と曲を先頭から再生成する。

### 7.1 無音環境を含む視覚プレイアビリティ

- レーン下端へ4区画のビートガイドを常時表示し、小節内の現在拍だけを点灯する。各4分音符の拍頭では判定線を120ms発光させる。
- ビートガイドと判定線発光は音声の有無に関係なく表示し、無音モードの選択操作やOS消音状態の検出を要求しない。
- ガイドは`AudioContext.currentTime`、`songStartTime`、BPMから現在拍を求め、音声・判定・表示ノートと同じ時計へ同期する。裏拍やスイングの入力時刻は追加表示しない。
- `prefers-reduced-motion: reduce`ではガイド区画の拡大アニメーションを無効化し、明度と輪郭だけで現在拍を示す。
- 音ありと無音で譜面、判定幅、得点、ダメージ、敵HPを変更しない。視覚ガイドは四分音符の基準だけを示し、ノートを読むゲーム性を維持する。
- ブラウザからOS音量、消音スイッチ、Bluetooth出力先の実際の可聴状態は確実に取得できないため、自動無音判定を前提としない。

## 8. 音声基準とヒント機能

- 判定、ヒント音、表示ノートは同じ譜面データの`note.time`を使用する。音声時刻を基準とし、表示から判定時刻を決めない。
- 「ストレート・バトル」は各4分音符頭のハイハットを強くし、ベース音も頭へ置く。8分音符裏には音程音を置かず、ヒントOFFでも基本譜面の拍を認識できる構成とする。
- ヒントは初期値ON。戦闘開始時の設定をその戦闘中に使用する。
- ヒントON時は、選択譜面の各`note.time`に25msのクリックと約110msの高いルート音を重ねる。ルート音は曲の和声から取得し、700Hz以上1400Hz以下へオクターブ移動する。基本、裏拍、技巧、余白、ブレイク技巧で共通に動作する。
- ヒント音源は通常音源と同じ管理対象とし、撃破、曲終了、再開始時に停止する。
- 撃破時はレーン中央に「撃破！」と最終スコア・コンボを表示し、次の戦闘開始時に消去する。
- スマートフォン向けミックスでは、ジャズベースのピークを0.20、ピアノを0.075とする。ベースは第2倍音36%、第3倍音12%を加え、700Hzまで通して小型スピーカーで輪郭を出す。
- 通常ヒントは三角波0.075、クリック0.045まで下げ、伴奏の奥でタイミングを示す。キャリブレーション音は三角波0.30、クリック0.24を維持する。
- マスターゲイン0.72は変更せず、他曲の音量とクリッピング余地を維持する。

## 9. PC版PoCの設定表示

- ヘッダー上段はPoC表記、タイトル、説明を横並びにし、タイトルは折り返さない。
- ヘッダー下段は曲、タップ、BPM、ヒント、戦闘開始を横一列にする。
- 説明文だけが残り幅を使用し、必要な場合に折り返す。幅560px以下では既存のスマートフォン表示を使用する。

## 10. 譜面別HPと時間切れ

- 敵最大HPは、選択譜面の全ノートをPERFECTした理論最大ダメージの70%を切り上げて決定する。
- スコア、PERFECT 18ダメージ、GOOD 10ダメージ、アクセント6追加は変更しない。
- PERFECTは1200点、GOODは600点とする。
- 基本、裏拍、技巧、余白は全PERFECT時に22拍目前後から撃破可能となる。
- ブレイク技巧は前半最大ダメージ420に対してHP429となり、長いブレイク後の復帰フレーズから撃破可能となる。
- 8小節終了時にHPが残っている場合は時間切れとし、レーンへ残りHPを表示する。

## 11. 端末別タイミング調整

- 攻撃入力は`pointerdown`で取得し、`event.timeStamp`から押下時のAudioContext時刻を逆算する。
- タイミング調整は2拍の練習後に8拍を計測し、中央値を±250ms以内で補正値とする。
- 補正値はバージョン付きで`localStorage`へ保存し、毎回の調整は求めない。保存できない環境では当該セッションだけ有効とする。
- 判定時は入力時刻から保存済み補正値を差し引く。
- 判定幅はPERFECT ±80ms、GOOD ±180msとする。
- 本体導入時も補正値はユーザーDBではなく端末側へ保存する。最近12入力以上が同方向へ40ms以上偏った場合だけ再調整を提案し、自動変更はしない。
- Bluetooth等の出力経路別プロファイルは将来拡張とする。

### 11.1 本体での能動的な再調整

- ユーザーはゲーム設定の「サウンド・タイミング調整」から、必要なときに任意でキャリブレーションを開始できる。
- 戦闘開始前の設定画面にも同じ導線を用意し、現在の補正値と最終調整日時を確認できるようにする。
- 「再調整」と「補正値をリセット」を分けて提供する。リセット後は補正値0msとして扱うのではなく、未調整状態へ戻す。
- 戦闘中に再調整を選んだ場合は、その場で開始せず、現在の戦闘を終了する確認を表示してから調整画面へ移動する。
- 能動的な再調整には回数制限を設けず、ログインやアプリ起動のたびには要求しない。

### 11.2 自動判定による再調整サジェスト

- 補正後の直近12件以上の有効タップを端末内で集計する。
- 中央値が同一方向へ40ms以上偏り、かつ75%以上の入力が同じ方向へ偏った場合に再調整候補とする。
- MISS、連打による重複入力、最寄りノートから±300msを超える入力は偏り判定から除外する。
- サジェストは戦闘中に割り込ませず、戦闘終了後に「タイミングのずれを検出しました。再調整しますか？」と表示する。
- 選択肢は「調整する」「あとで」とし、補正値を自動更新しない。
- 「あとで」を選んだ場合は同一端末で7日間再表示しない。明示的な設定画面からの再調整は抑制期間中も利用できる。
- サジェスト判定データ、抑制期限、補正値は端末ローカルに保持し、ユーザーDBへ同期しない。

### 12.1 Android Chrome時計診断

診断表示はURLへ`?debug=1`を付けた場合だけ有効になる。公開検証URLは次のとおり。

`https://gerupon-lgtm.github.io/beat-poc/?debug=1`

同一端末、同一曲、同一譜面、同一BPM、同一ヒント設定で次の順に比較する。

1. 通常版Chrome、音量あり
2. 通常版Chrome、端末音量0。URLを再読み込みしてから開始する。
3. LINEアプリ内ブラウザ、端末音量0。URLを再読み込みしてから開始する。

プレイ中は診断値をメモリ内にだけ集計し、診断パネルへのDOM書き換えは行わない。勝利または時間切れの後に最終サマリーが一度だけ表示されるため、プレイ終了後に無理なくスクリーンショットできる。特に`audioSec`、`wallSec`、`drift`、`maxAbs`、`frameMax`、`>50`、`renderMax`、`>8`、`timerMax`、`>75`、`audio`、`states`、`running`、`tap`を比較する。

- `drift`だけが正方向へ増加: 音声時計が実時間より遅れている。
- `drift`は小さく、`frameMax`と`>50`が増加: メインスレッドの描画停止が発生している。
- `audio`が`running`以外へ変化: AudioContextの状態変化が発生している。
- `running=false`の時点で`audioSec`が曲長未満: 壁時計の終了タイマーが音声時計より先に終了させている。

追加診断では、`renderMax`が1フレームの描画処理時間、`timerMax`が25ms周期の音声スケジューラー呼び出し間隔を示す。

- `frameMax`だけが大きく、`renderMax`と`timerMax`が小さい: Chromeが描画コールバックだけを間引いている可能性が高い。
- `frameMax`と`timerMax`が大きく、`renderMax`が小さい: メインスレッド全体またはタイマー配送が停止している。
- `renderMax`と`>8`も大きい: PoC自身のDOM・レイアウト・描画処理が主因候補になる。

実機診断では、Android Chromeで`frameMax`と`>50`だけが増え、`render>8=0`かつ`timer>75=0`だった。この結果から、PoCの描画処理やメインスレッド全体ではなく、rAFコールバックの間引きが主因候補になった。

JavaScript `v=20`では、通常のrAF描画を維持しながら、表示中の戦闘で最後の視覚描画から50ms以上空いた場合だけ25ms周期のウォッチドッグが補助描画する。rAFが正常な環境では補助DOM描画を行わず、非表示タブと戦闘終了後では動作しない。診断サマリーの`fallback`は補助描画の発動回数を示す。音声時計、判定、譜面、スコア、曲終了処理は変更していない。

JavaScript `v=21`では、HITまたはMISSが確定したノーツをDOMと内部Mapの両方から削除し、撃破・時間切れ・再開始時には残りノーツを全消去する。これにより、MISSしたノーツが最後の位置に停止して4区画ビートガイドとずれて見える問題を修正した。

JavaScript `v=22`では、v20で追加した25ms周期のrAFウォッチドッグを撤回した。実機検証で補助描画が発動していない環境でもタップ後の操作感が悪化し、Android Chrome音量0の問題も安定して解消しなかったためである。描画はrAFだけに戻し、v21の停止ノーツ削除と低負荷診断は維持する。

JavaScript `v=23`では、通常URLのrAF表示を維持したまま、`?visual=compositor`で落下ノーツと4区画ビートガイドをWeb Animations APIのtransform・opacity合成アニメーションへ切り替えられる。ノーツとガイドは同じ曲開始アンカーを使用する。非対応環境では自動的にrAF表示へ戻る。これはPoCのA/B検証用であり、プレイヤー向け設定ではない。

JavaScript `v=24`では、合成アニメーション版で欠けていた4区画ビートガイドの黄色発光とTAP判定ラインの拍頭発光を、専用の静的発光レイヤーに対するtransform・opacityアニメーションとして復元した。音声時計、判定、ノーツ移動、通常URLのrAF表示は変更しない。

比較URL:

- rAF基準版: `https://gerupon-lgtm.github.io/beat-poc/`
- 合成アニメーション版: `https://gerupon-lgtm.github.io/beat-poc/?visual=compositor`
- 合成版の診断併用: `https://gerupon-lgtm.github.io/beat-poc/?visual=compositor&debug=1`

変更前のv22実行一式は`dist/archive/rhythm-battle-poc-v22-raf-baseline-20260620.zip`へ保存した。SHA-256は`CE56EAB83CAA8CB2AB07BDDEA697C7B1C3F808B85242132FF5BC9845C0DAB513`。復元時はZIP内の`index.html`、`css/`、`js/`、`README.md`を公開先へ上書きする。

診断値は終了後の画面表示だけに使用し、外部送信・永続保存は行わない。通常URLでは診断計測もパネル表示も行わない。

## 13. 実装済みと本体未実装の区別

### PoCで実装済み

- 5曲・5譜面の任意組み合わせ、BPM 80～180、4拍カウント
- 音声、判定、表示ノート、ヒント、視覚ガイドの共通時計同期
- `pointerdown`と`event.timeStamp`を使う入力時刻補正
- PERFECT `±80ms`、GOOD `±180ms`、MISS時に次ノートを消費しない判定
- 動的敵HP、撃破・時間切れ、8小節で停止する再生ライフサイクル
- 任意実行できる端末別キャリブレーションと`localStorage`保存
- 4区画ビートガイド、拍頭発光、全ノーツ22px、低モーション対応
- `?debug=1`で有効になり、プレイ中はメモリ集計だけを行って終了後に一度だけ表示する低負荷診断
- HIT・MISS・撃破時に停止ノーツを残さない表示寿命管理
- URLでA/B比較できる合成アニメーション表示モード

### ゲーム本体向け方針のみ・未実装

- ゲーム設定と戦闘開始前設定の両方から行う能動的な再調整
- 入力偏りの自動検出と、戦闘終了後の再調整サジェスト
- 敵、勝敗、経験値、アイテム処理への接続
- 楽曲・譜面データの本体用形式と難易度段階
- Bluetoothなど出力経路別の補正プロファイル
